3D point cloud generation with frame fusion
==========================================

This project builds coloured 3D point clouds from (1) a real start photograph,
(2) a RealEstate10K-style camera trajectory, and (3) a generated video that
follows that trajectory. The implementation lives in generate_pointclouds.py
at the repository root; outputs are written under this folder (pointclouds/).


What files are wired together
-----------------------------

For each video in generated_videos/:

  - Name pattern:  <base>_<n>.mp4   or   <base>_standard.mp4

  - Start image:   start_images/<base>.jpg

  - Trajectory:    trajectories/<base>_<n>.txt          for variant _1, _2, ...
                   trajectories/<base>.txt             for _standard

  - Output cloud:  pointclouds/<same_stem_as_video>.ply

Example:  generated_videos/abc123_2.mp4  uses  start_images/abc123.jpg,
trajectories/abc123_2.txt, and writes  pointclouds/abc123_2.ply.


Why “fusion”
------------

A cloud built only from the start image and the first pose is the same for
every trajectory variant of a scene, because those trajectories share the same
starting camera row. To show how different camera paths (and different
generated videos) change the 3D reconstruction, we merge geometry from many
time steps along the path: the high-quality start view plus several frames
from the generated video, each unprojected with its own extrinsics. That
process is what we call fusion here.


Depth model
-----------

  - Primary:  Depth Anything V2 Metric Indoor (Hugging Face), giving depth in
    metres on indoor content.

  - Fallback: Depth Anything V2 Large (relative); used only if the metric
    model cannot be loaded.

  - Device:   MPS, CUDA, or CPU (automatic).


Per-video fused pipeline (step by step)
---------------------------------------

1. Load trajectory

   The .txt file is parsed as RealEstate10K: normalised intrinsics
   (fx, fy, cx, cy) and world-to-camera 4x4 matrices per frame. The first pose
   corresponds to the start image viewpoint.

2. Start image (high resolution)

   Run the depth model once on start_images/<base>.jpg at native resolution.
   Depth is cached per <base> so all five videos for the same scene reuse one
   HR depth map (four scenes → four HR depth runs).

   Pixels with depth outside [D_MIN, D_MAX] metres are discarded (defaults:
   0.15 m to 6.0 m). Remaining pixels are unprojected to 3D using the mean
   normalised intrinsics scaled to the image width and height, and the inverse
   of the first world-to-camera matrix (camera-to-world at the start).

3. Video frames (generated pixels)

   The clip is decoded with OpenCV. The number of frames in the video may
   differ from the number of poses in the trajectory.

   Frames are aligned to poses the same way as in gaussian_splatting_from_
   image_trajectory.ipynb: for each of min(video_frames, poses) steps, pair
   one video frame index and one pose index via evenly spaced linspace over
   their respective index ranges.

   From that full pairing, a fixed number of samples are taken (default:
   N_FRAMES_PER_VIDEO = 12), again evenly spaced along the paired sequence.

   For each sampled frame:

     - Read RGB at video resolution.
     - Run the depth model on that frame.
     - Unproject valid pixels using intrinsics scaled to that frame’s width and
       height and the camera-to-world matrix for the paired pose index.

4. Merge

   3D points and RGB colours from the start image and from all sampled video
   frames are concatenated in one array.

5. Post-process (Open3D)

   - Voxel downsampling (default 0.015 m for fused clouds) so overlap across
     views agrees in one voxel while inconsistent geometry often remains as
     separate points or thicker regions.

   - Statistical outlier removal (neighbour count and std threshold tuned to
     be relatively loose for fused clouds so trajectory failure modes stay
     visible).

   - Normal estimation and orientation toward the first camera centre.

6. Save

   Binary PLY:  pointclouds/<video_stem>.ply


Anchor clouds (not fused along the video)
-----------------------------------------

Once per scene, the script also writes  pointclouds/<base>_anchor.ply :

  - Built only from the start image and its depth (same HR depth as above).

  - Uses the first pose of the trajectory file encountered when that scene
    is first processed (in practice, first-pose rows match across variant .txt
    files for these datasets).

  - Tighter voxel size (default 0.01 m) and stricter outlier filtering than the
    fused clouds.

This file is a compact, photometric reference: “one real photo in 3D,” without
mixing in synthetic video frames.


Important limitations
-----------------------

  - Depth on generated video frames can be wrong or inconsistent with the
    poses; fusion then shows drift, ghosting, or rare extreme outliers.

  - Intrinsics are averaged over the trajectory per frame set; all sampled
    video frames share one focal length and principal point in pixels for their
    resolution.

  - Metric depth and RE10K poses are not guaranteed to be in the same global
    scale; the pipeline follows the notebook and trusts the metric indoor
    model where available.


Re-running or tuning
--------------------

  Edit the constants at the top of generate_pointclouds.py:

    N_FRAMES_PER_VIDEO   — more frames → denser fusion, longer runtime
    D_MIN / D_MAX        — depth keep range (metres)
    VOXEL_SIZE_FUSED     — fused cloud resolution
    OUTLIER_STD_FUSED    — higher → keep more stray points (more drift visible)

  Run:  .venv/bin/python generate_pointclouds.py

  (Use the project virtualenv so torch, open3d, and transformers match
   requirements.txt.)
